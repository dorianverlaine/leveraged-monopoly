# Internationalization (i18n) contract

The frontend ships in four languages. This document is the contract between the
backend (which is language-neutral) and the frontend translation layer.

**Supported locales** (`monopoly.accounts.store.SUPPORTED_LOCALES`):

| code      | language                     |
|-----------|------------------------------|
| `en`      | English (default / fallback) |
| `zh-Hant` | 繁體中文 (Traditional Chinese) |
| `zh-Hans` | 简体中文 (Simplified Chinese)  |
| `fr`      | Français (French)            |

The player's chosen locale is stored on their account and returned in the
account profile (`locale`), so a returning client restores its language
automatically. Anonymous guests fall back to `en` (or the browser locale, the
frontend's choice).

## The one rule: translate by code, never display server English

The backend emits **stable machine codes**, and English strings only as a
developer-facing fallback. The frontend must key its translations off the codes
and must **never render the server's English `message` / `note` fields** to end
users. This keeps every user-visible string translatable and the four locales in
sync.

- Rule rejections: use `RuleError.code` (not `.message`).
- Drama / ledger events: use the ledger entry's `kind` (not `.note`).
- Action buttons: use the `ActionType` string.

The English `message` / `note` fields stay useful for logs, tests, and debugging.

## Keys to translate

### 1. Action types (control-panel buttons) — `ActionType`
`roll_dice`, `buy`, `mortgage`, `unmortgage`, `leverage`, `repay_debt`,
`securitize`, `end_turn`, `concede`

The `available` array in each `state` message tells the client which of these to
enable right now.

### 2. Rule-error codes — `RuleErrorCode`
`not_your_turn`, `wrong_phase`, `game_over`, `unknown_action`,
`player_not_active`, `player_bankrupt`, `insufficient_cash`,
`insufficient_collateral`, `over_borrow`, `nothing_to_repay`,
`tile_not_purchasable`, `tile_already_owned`, `not_tile_owner`,
`tile_already_mortgaged`, `tile_not_mortgaged`, `invalid_target`,
`invalid_amount`, `invalid_percent`

Plus transport-level error codes from the server: `bad_frame`, `unknown_type`,
`no_such_room`, `cannot_join`, `room_full`, `no_room`, `not_playing`,
`not_seated`, `bad_token`, `already_started`, `not_host`, `no_accounts`,
`bad_session`, `bad_device_key`.

### 3. Ledger / event kinds (the drama feed) — `Transaction.kind`
`salary`, `tax`, `rent`, `buy`, `leverage`, `repay`, `mortgage`, `unmortgage`,
`securitize`, `liquidation`, `bankruptcy`, `interest`, `shock`

Each ledger entry also carries structured data (`player_id`, `amount`,
`round_number`) so a translated template can interpolate values, e.g.
`rent` → `en: "{player} paid {amount} rent"` / `zh-Hant: "{player} 支付了 {amount} 租金"`.

> **Note / future work:** event `note` strings are currently pre-formatted
> English that sometimes embed a tile name. For fully localized drama popups the
> events should carry the tile index / name as a structured field rather than
> inside the note. Tracked as a follow-up; `kind` + `amount` + `player_id` are
> enough for a first localized version.

### 4. Board / tile keys — `Tile.key`
Every tile carries a stable key the frontend localises (rather than showing the
English `name`):

- Cities (groups): `hong_kong`, `paris`, `new_york`.
- Landmarks: `<city>:<slug>`, e.g. `hong_kong:central`, `paris:la_defense`,
  `new_york:wall_street`. The English `name` is the fallback and the slug source.
- Structural tiles: `go`, `wealth_tax`, `vacation`, `marketplace`, `event`.

City display names (the group labels):

| key | en | zh-Hant | zh-Hans | fr |
|-----|-----|---------|---------|-----|
| `hong_kong` | Hong Kong | 香港 | 香港 | Hong Kong |
| `paris` | Paris | 巴黎 | 巴黎 | Paris |
| `new_york` | New York | 紐約 | 纽约 | New York |

Landmark names (36 keys total) should be localised too; many are proper nouns
that stay close across languages (e.g. `new_york:wall_street` → 華爾街 / 华尔街 /
Wall Street). The frontend owns the full landmark catalogue.

### 5. UI string keys (frontend-owned)
Not emitted by the backend, but part of the same catalogue the frontend
maintains: lobby / matchmaking, profile & progression (level, XP, rating,
streak), and the drama overlays. Suggested namespaces: `lobby.*`, `game.*`,
`profile.*`, `event.*`, `error.*`, `action.*`.
